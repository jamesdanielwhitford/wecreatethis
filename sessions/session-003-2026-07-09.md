# Session 003 - 2026-07-09

## What was done

Merged Hardle and Randle into a single page/URL on branch `hardle-update` (pushed, not yet merged to main), redesigned around two references: the user's native iOS Hardle rebuild (dot-based scoring, no Easy Mode) and Zanagrams (results screen, hamburger menu UX).

### 1. Single page, mode as state
- Collapsed `index.html` + `randle.html` into one `index.html`. Deleted `randle.html`, `testle.html`, and their unique icon assets.
- Replaced all `window.location.pathname.includes('randle'/'testle')` branching in `app.js`/`ui.js`/`game.js` with a `game.variant` field (`'hardle' | 'randle'`).
- Switching Hardle <-> Practice Mode (Randle) is now in-place (no navigation/reload) via `handleSwitchToRandle()` / `handleSwitchToHardle()` in `app.js`.
- Homepage (`WeCreateThis/index.html`) no longer has a separate Randle icon, just one Hardle tile.
- `sw.js` bumped to v57, `ASSETS` trimmed to match.

### 2. Removed Easy Mode
- Deleted all auto-deduction logic from `game.js` (`deduceTiles`, `updateKeyboardFromDeduction`, `getLetterPositions`, `toggleMode`) and the mode-selection UI from `app.js`/`index.html`.
- Manual tile-marking (tap once = red, tap again = green, tap again = clear) is now always on, no mode toggle anywhere.

### 3. Dot-based scoring (replaces numeric score badge)
- `ui.js` now renders a 2x2 dot cluster per row (`score` dots green, `4-score` red, gray placeholders pre-submit) instead of a number-in-circle. `calculateScore()` itself was unchanged (already 0-4 count-only).
- Share text emoji grid updated to match (green/red squares instead of yellow).

### 4. Zanagrams-style results screen
- New `#end-game-modal` layout: big attempts stat, status line, Share Score (green) + Practice Mode/Play Again (blue) buttons.
- Originally included a mini per-guess board recap (word + dots); user asked to remove it after seeing it live, it looked bad. Removed entirely - results screen is just the stat/status/buttons now.

### 5. Hamburger menu
- Replaced the old info-icon + quick-settings-icon pair with one hamburger button opening a Zanagrams-"Paused"-style menu: Practice Mode/Daily Puzzle toggle (top), How to Play, Send Feedback.
- Send Feedback opens an in-modal sub-view (not a bare `mailto:` link) showing `james@wecreatethis.com` as selectable text, a Copy Email button, and an Open in Mail button.
- How to Play was fully rewritten as a compact single-screen legend (4 rows: partial match, all-red/no-match, tile-marking, all-green/win), replacing the old long per-example paragraph format that didn't fit in one modal.

### Polish pass (post-implementation, from live testing)
- Desktop navbar constrained to max-width 600px instead of spanning full viewport; removed its border entirely (both side and bottom).
- Enter key (checkmark) colored green, Backspace key (X) colored red on the keyboard.
- Share Score button green, Practice Mode button blue.
- "Got it!" button glow/shadow simplified to match Share Score's flat style (no box-shadow/hover-lift).
- All "Randle" user-facing text replaced with "Practice Mode" (nav title, share text, PWA manifest); internal code (`variant: 'randle'`, localStorage keys, function names) left as-is since it's not user-visible.

## Decisions and reasoning

- **Full merge into one URL** (not just unlinking Randle from the homepage) - user explicitly chose this over keeping `randle.html` as a standalone page, so Practice Mode has no separate route at all now.
- **Mailto feedback link â†’ in-modal email view**: a bare `mailto:` does nothing visible without a configured mail client; showing the address as copyable text plus both a copy and an open-in-mail action covers desktop users without one.
- **Dropped the mini board recap**: it was explicitly requested in the original plan (mirroring Zanagrams' visual summary) but looked bad in practice once built and shown; user asked to cut it rather than restyle it.
- Found and ruled out a suspected "wrong row shakes" bug - traced the logic, determined `shakeRow(game.currentRow)` is only called pre-increment on invalid words, so it's provably correct; the original report was very likely a stale service-worker cache during testing (hit that same staleness issue repeatedly this session, had to unregister the SW to see any change).
